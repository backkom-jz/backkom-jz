# 高级 PHP 工程师面试复习计划与面试解析

## 目标定位

高级 PHP 工程师面试不只考 PHP 语法，更看重以下能力：

- 能解释 PHP 运行机制、FPM 模型、内存管理、自动加载、异常和错误处理。
- 能设计稳定的 Web 后端架构，包括接口、缓存、数据库、队列、日志、监控和部署。
- 能处理高并发、慢 SQL、缓存击穿、消息堆积、接口超时、线上故障。
- 能讲清楚自己做过的项目，而不是只背概念。
- 能从业务、工程、性能、稳定性、安全性几个维度分析问题。

## 一周复习安排

### Day 1：PHP 基础进阶与运行机制

复习重点：

- PHP 变量、引用、类型转换、数组底层特点。
- `isset()`、`empty()`、`is_null()` 区别。
- `==` 与 `===` 区别。
- `include`、`require`、`include_once`、`require_once` 区别。
- `trait`、`interface`、`abstract class` 使用场景。
- Composer 自动加载与 PSR-4。
- PHP-FPM 工作模型。
- OPcache 作用。

高频解析：

**PHP 数组为什么既能当数组又能当字典？**

PHP 的 array 本质是有序 HashTable，既支持整数 key，也支持字符串 key。优点是灵活，缺点是内存占用比普通 C 数组更高。所以在大数据量场景下，PHP array 很方便但不省内存。

**PHP-FPM 是什么？**

PHP-FPM 是 FastCGI 进程管理器。Nginx 接收 HTTP 请求后，把 PHP 请求转给 PHP-FPM worker 处理。FPM 通过 master 管理多个 worker 进程，请求结束后 worker 继续复用。

**OPcache 有什么用？**

PHP 每次执行脚本前都要解析源码并生成 opcode。OPcache 会缓存编译后的 opcode，减少重复编译成本，提高性能。

### Day 2：面向对象、设计模式与工程规范

复习重点：

- 面向对象三大特性：封装、继承、多态。
- SOLID 原则。
- 工厂、单例、策略、观察者、装饰器、责任链。
- 依赖注入与控制反转。
- PSR-1、PSR-4、PSR-12。
- Composer 包管理。
- 领域分层：Controller、Service、Repository、Model。

高频解析：

**接口和抽象类怎么选？**

接口强调行为契约，适合定义能力，比如 `CacheInterface`、`LoggerInterface`。抽象类可以放公共属性和公共方法，适合多个子类共享部分实现。业务开发中，如果只是约定能力，优先使用接口。

**策略模式适合什么场景？**

当一个业务有多种可替换算法或规则时使用。例如支付渠道、优惠券计算、物流计费、风控规则。策略模式可以减少大量 `if else`，让新增规则通过新增类实现。

**Repository 层有什么价值？**

Repository 用来隔离业务逻辑和数据访问。Service 不直接关心 SQL、ORM 或缓存细节，便于测试和后续替换存储方案。

### Day 3：MySQL 面试重点

复习重点：

- InnoDB 存储引擎。
- B+Tree 索引。
- 聚簇索引、二级索引、覆盖索引、联合索引。
- 最左前缀原则。
- explain 分析。
- 慢 SQL 优化。
- 事务 ACID。
- MVCC。
- 间隙锁、行锁、表锁。
- 主从复制、读写分离、分库分表。

高频解析：

**为什么 MySQL 索引用 B+Tree？**

B+Tree 层级低，磁盘 IO 次数少。非叶子节点只存 key，叶子节点存完整数据或主键，并且叶子节点有链表，适合范围查询。

**联合索引为什么要遵守最左前缀？**

联合索引按定义顺序排序，例如 `(a,b,c)` 会先按 `a` 排，再按 `b`，再按 `c`。如果查询跳过 `a`，后面的有序性无法直接利用。

**慢 SQL 怎么排查？**

先看慢日志，再用 `explain` 分析执行计划。重点看是否走索引、扫描行数、是否 filesort、是否 temporary。优化方向包括加索引、改 SQL、减少返回字段、分页优化、拆表、缓存热点数据。

**事务隔离级别有哪些？**

- 读未提交：可能脏读。
- 读已提交：解决脏读，可能不可重复读。
- 可重复读：MySQL InnoDB 默认级别，配合 MVCC 和锁解决大部分问题。
- 串行化：最严格，性能最低。

### Day 4：Redis、缓存与分布式锁

复习重点：

- Redis 数据结构：String、Hash、List、Set、ZSet、Bitmap、HyperLogLog、Stream。
- 缓存穿透、击穿、雪崩。
- Redis 过期策略。
- Redis 内存淘汰策略。
- Redis 持久化：RDB、AOF。
- Redis 主从、哨兵、Cluster。
- 分布式锁。
- 延迟队列、排行榜、计数器、限流。

高频解析：

**缓存穿透、击穿、雪崩区别？**

- 穿透：查询不存在的数据，请求直接打到数据库。解决：缓存空值、布隆过滤器。
- 击穿：热点 key 过期，大量请求同时打到数据库。解决：互斥锁、热点 key 不过期、逻辑过期。
- 雪崩：大量 key 同时过期或 Redis 故障。解决：过期时间加随机值、多级缓存、限流降级。

**Redis 分布式锁要注意什么？**

加锁要设置唯一 value 和过期时间，释放锁时必须校验 value，避免误删别人的锁。释放锁建议用 Lua 脚本保证原子性。业务执行时间可能超过锁过期时间时，需要考虑续期或幂等补偿。

**Redis 为什么快？**

主要因为基于内存、单线程避免频繁上下文切换、IO 多路复用、高效数据结构。Redis 6 后引入多线程主要用于网络 IO，命令执行仍以单线程模型为核心。

### Day 5：高并发、队列与系统设计

复习重点：

- 限流、降级、熔断。
- 消息队列：RabbitMQ、Kafka、Redis Stream。
- 异步化。
- 幂等设计。
- 分布式事务。
- 最终一致性。
- 订单、支付、库存、秒杀系统。
- 接口防重复提交。

高频解析：

**如何设计秒杀系统？**

核心思路是削峰、限流、异步化和库存保护。

常见方案：

- 活动页面静态化，减少动态请求。
- Redis 预扣库存。
- 接口限流，防止瞬时流量打垮服务。
- 请求进入消息队列异步下单。
- 数据库最终扣减库存。
- 下单接口做幂等，避免重复下单。

**消息队列如何保证不丢消息？**

生产端开启 confirm，Broker 开启持久化，消费端手动 ack。消费失败时重试或进入死信队列。业务侧必须做幂等，因为消息可能重复投递。

**如何设计接口幂等？**

常见方案：

- 唯一请求号。
- Token 防重复提交。
- 数据库唯一索引。
- 状态机控制。
- Redis `setnx`。

核心原则是同一个业务请求执行一次和执行多次，最终结果一致。

### Day 6：Laravel / Hyperf / Swoole 专项

复习重点：

- Laravel 生命周期。
- Service Container 服务容器。
- Service Provider 服务提供者。
- Middleware 中间件。
- Eloquent ORM。
- Queue、Event、Listener、Job。
- Swoole 常驻内存模型。
- 协程。
- Hyperf 与传统 FPM 的区别。

高频解析：

**Laravel 一次请求生命周期是什么？**

入口是 `public/index.php`，加载 Composer，创建应用实例，绑定 Kernel，请求进入 HTTP Kernel，经过中间件、路由、控制器、业务逻辑，生成 Response 后返回，最后执行 terminate 逻辑。

**Laravel 服务容器解决什么问题？**

服务容器负责对象创建和依赖管理。通过依赖注入，可以降低类之间耦合，便于测试、替换实现和管理生命周期。

**Swoole 和 PHP-FPM 最大区别是什么？**

PHP-FPM 是请求结束后释放上下文的短生命周期模型。Swoole 是常驻内存模型，进程长期运行，适合高并发和长连接，但要特别注意全局变量污染、内存泄漏、连接复用和协程安全。

**协程要注意什么？**

协程是用户态轻量级调度。使用协程时要避免阻塞操作，数据库、Redis、HTTP 客户端都要使用协程版客户端。共享变量要注意并发安全。

### Day 7：线上排障、安全与综合模拟面试

复习重点：

- 日志体系。
- 链路追踪。
- 慢请求排查。
- CPU 飙高、内存泄漏、连接耗尽。
- Nginx、PHP-FPM、MySQL、Redis 常见故障。
- SQL 注入、XSS、CSRF。
- 鉴权、权限、签名、防重放。
- 项目复盘表达。

高频解析：

**接口突然变慢怎么排查？**

先确认影响范围，是单接口、单机器还是全站。然后看监控：QPS、响应时间、错误率、CPU、内存、网络、磁盘。再查日志和链路追踪，定位耗时在 PHP、MySQL、Redis、第三方接口还是队列。最后针对具体瓶颈处理，比如 SQL 优化、缓存、扩容、降级或熔断。

**PHP-FPM 连接打满怎么办？**

先看 `pm.max_children` 是否耗尽，再看慢日志和请求耗时。可能原因包括慢 SQL、外部接口阻塞、死循环、锁等待、Redis/MySQL 连接慢。短期可以扩容 worker 或降级流量，长期要优化慢请求和超时控制。

**如何防 SQL 注入？**

使用 PDO 预处理或 ORM 参数绑定，不拼接用户输入。对输入做类型校验，对输出做转义。数据库账号权限最小化。

**如何防接口重放？**

常见方案是 timestamp + nonce + sign。服务端校验时间窗口，nonce 只能使用一次，签名使用双方约定密钥生成。敏感接口还要配合 HTTPS、用户鉴权和操作审计。
