# 高级后端面试准备手册

> 适用目标：高级 PHP / 后端工程师，覆盖 PHP 内核与框架、MySQL、Redis、RabbitMQ、Go、容器与运维、系统设计、高并发分布式、项目深挖、PMP 与技术管理。

## 使用方式

面试准备不要只背概念，建议按“三层回答法”准备：

1. 先给结论：一句话回答核心问题。
2. 再讲机制：解释底层原理、关键链路和边界条件。
3. 最后落项目：结合 DCT、WMS、TMS、中台、积分、库存、提现、消息中心等项目说你怎么做、怎么排障、怎么取舍。

回答高级岗位问题时，优先体现四种能力：

- 能定位问题：慢 SQL、连接耗尽、缓存穿透、消息堆积、CPU 飙高。
- 能做取舍：一致性与性能、顺序性与吞吐、强事务与最终一致性。
- 能落方案：分层、幂等、重试、补偿、灰度、监控、告警。
- 能讲项目：背景、复杂点、方案、数据、故障、复盘。

## 一、PHP 内核与运行机制

### 1. 请求生命周期

Nginx 不执行 PHP，它接收请求后通过 FastCGI 协议转发给 PHP-FPM。PHP-FPM master 管理 worker，worker 加载脚本，OPcache 命中则直接执行 opcode，未命中则编译后执行，请求结束后释放大部分请求级变量。

面试展开：

- PHP-FPM 是多进程模型，不是每个请求新启动一个 PHP 解释器。
- `pm.max_children` 决定最大并发 worker，打满后请求排队或超时。
- `pm.max_requests` 可让 worker 定期重启，缓解扩展或业务代码导致的内存泄漏。
- OPcache 缓存 opcode，减少语法解析和编译开销。

### 2. PHP 变量与 zval

PHP 变量名存在符号表中，变量值由 `zval` 表示。`zval` 包含类型、值、引用计数等信息。PHP 使用写时复制，多个变量指向同一份值时不会立即复制，只有发生写入且引用计数大于 1 时才分离。

高频问法：

- **PHP 引用和写时复制有什么区别？**  
  普通赋值共享底层值，写入时复制；引用赋值让两个变量指向同一个引用容器，修改任意一个都会影响另一个。

- **为什么 PHP 数组占内存大？**  
  PHP array 是有序 HashTable，同时支持整数 key 和字符串 key，还要维护桶、哈希、插入顺序等元信息，灵活但内存开销高。

### 3. 垃圾回收

PHP 主要依赖引用计数回收变量。当出现循环引用时，单纯引用计数无法释放，需要 GC 根缓冲区扫描可能的循环垃圾。

面试回答：

- 普通变量离开作用域后引用计数减一，降到 0 即释放。
- 循环引用会导致引用计数不为 0，需要周期性 GC 处理。
- 长生命周期进程如 Swoole、Worker、队列消费者要特别关注静态变量、全局数组、闭包引用、容器单例造成的内存增长。

### 4. PHP-FPM 调优

核心参数：

- `pm.max_children`：最大 worker 数，受 CPU、内存、单请求内存占用影响。
- `pm.start_servers`、`pm.min_spare_servers`、`pm.max_spare_servers`：动态模式下 worker 数量。
- `request_slowlog_timeout`：慢请求日志阈值。
- `max_execution_time`：脚本最大执行时间。
- `request_terminate_timeout`：FPM 层强制终止超时请求。

排查连接打满：

1. 看 FPM 状态页，确认 active、idle、listen queue。
2. 查 slowlog，定位阻塞函数或业务代码。
3. 查 MySQL 慢 SQL、Redis 慢日志、外部 HTTP 超时。
4. 短期扩容 worker，长期优化慢链路、加超时、异步化。

### 5. Composer 与自动加载

Composer 通过 `vendor/autoload.php` 注册自动加载器。PSR-4 根据命名空间前缀映射目录路径，使用类时再加载对应文件。生产环境建议 `composer dump-autoload -o` 生成优化 classmap。

### 6. 错误、异常与工程规范

- 业务异常：参数错误、状态不允许、余额不足，可转成明确错误码。
- 系统异常：数据库异常、Redis 异常、第三方超时，需要日志、告警、降级。
- 不要吞异常；要记录上下文，包括 trace id、user id、order id、请求参数摘要。
- Controller 不写复杂业务；Service 编排流程；Repository 隔离数据访问；Job/Event 处理异步任务。

## 二、Laravel、Yii2、CodeIgniter 4 框架源码

### 1. Laravel 请求生命周期

入口是 `public/index.php`，加载 Composer，创建 Application，绑定 HTTP Kernel，请求经过全局中间件、路由匹配、路由中间件、控制器、业务逻辑，生成 Response 返回，最后执行 terminate。

面试重点：

- Service Container 管对象创建和依赖注入。
- Service Provider 负责注册和启动服务。
- Facade 是静态代理，底层仍从容器解析对象。
- Middleware 适合鉴权、限流、日志、跨域、请求上下文。
- Eloquent 是 Active Record，开发效率高，但复杂查询要注意 N+1、隐式查询和大对象开销。

### 2. Laravel 容器

容器解决“对象如何创建、依赖如何装配、生命周期如何管理”的问题。

常见绑定：

```php
$this->app->bind(PaymentInterface::class, WechatPayment::class);
$this->app->singleton(ConfigRepository::class, fn () => new ConfigRepository());
```

回答方式：

- `bind` 每次解析新实例。
- `singleton` 全局复用同一实例。
- 依赖接口而不是依赖具体类，方便替换实现和单元测试。
- 容器不是为了省 `new`，而是为了管理复杂依赖图。

### 3. Laravel 事件、队列与任务

事件适合业务解耦，例如订单支付成功后发积分、发通知、同步报表。队列适合耗时、可重试、非强实时任务。Job 要设计幂等，避免重试导致重复发券、重复扣款、重复通知。

### 4. Yii2 核心机制

Yii2 的入口通常是 `web/index.php`，加载配置并创建 `yii\web\Application`。Application 负责请求解析、路由分发、控制器执行和响应返回。

面试重点：

- Component + Behavior：组件提供基础能力，行为用于横向扩展对象功能。
- DI Container：支持依赖注入和对象配置。
- ActiveRecord：一张表对应一个 AR 类，适合 CRUD，但复杂业务建议 Service 层封装。
- Filter：控制器前后置过滤，适合鉴权、访问控制、动词限制。

Yii2 和 Laravel 对比：

- Laravel 生态和语法糖更强，容器、队列、事件体系成熟。
- Yii2 配置化、Gii、AR、表单验证能力强，传统后台和管理系统效率高。
- 高级面试不要争谁更好，要说“根据团队生态、项目复杂度、长期维护成本选择”。

### 5. CodeIgniter 4 核心机制

CI4 更轻量，入口 `public/index.php` 启动框架，路由匹配 Controller，Controller 调用 Model 或 Service，返回 Response。

面试重点：

- 轻量、上手快、约束少，适合中小项目和快速交付。
- Service 类可统一创建核心服务。
- Filter 类似中间件，用于认证、CSRF、限流、日志。
- Model 封装表访问，复杂业务仍应放到 Service。

### 6. 框架源码类问题回答模板

问题：“你看过 Laravel/Yii/CI 源码吗？”

回答：

```text
我重点看过请求启动、路由分发、容器解析、中间件执行和 ORM 查询构建部分。
以 Laravel 为例，请求从 public/index.php 进入，创建 Application 和 Kernel，
经过 middleware pipeline，路由匹配到 Controller，依赖由容器反射解析，
最后返回 Response。看源码的价值是理解框架边界：什么适合放中间件，
什么适合放 Service，什么逻辑不要塞进 Model 或 Controller。
```

## 三、MySQL 从底层到调优

### 1. 核心知识图谱

- SQL 执行流程：连接器、解析器、预处理器、优化器、执行器、存储引擎。
- InnoDB 存储：表空间、段、区、页、行。
- 索引：B+Tree、聚簇索引、二级索引、联合索引、覆盖索引、索引下推。
- 事务：ACID、隔离级别、MVCC、undo log、redo log、binlog。
- 锁：行锁、间隙锁、next-key lock、意向锁、元数据锁。
- 调优：慢日志、EXPLAIN、索引设计、SQL 改写、分页优化、分库分表。

### 2. 慢 SQL 排查模板

1. 从监控确认慢接口、慢 SQL、QPS、连接数、CPU、IO。
2. 开慢查询日志，拿到 SQL、执行时间、扫描行数。
3. 用 `EXPLAIN` 看 type、key、rows、Extra。
4. 判断是否索引失效、回表过多、排序临时表、锁等待。
5. 优化索引、改写 SQL、减少字段、延迟关联、游标分页。
6. 验证执行计划和线上指标，避免只在测试库看结果。

### 3. MySQL 100+ 高频面试题速查

#### 基础与执行流程

1. **SQL 在 MySQL 中怎么执行？**  
   连接认证后经过解析、预处理、优化、执行，执行器调用 InnoDB 等存储引擎读取或修改数据。

2. **MySQL 8 为什么移除查询缓存？**  
   表更新会导致相关缓存失效，高写入场景命中率低且维护成本高。

3. **InnoDB 和 MyISAM 区别？**  
   InnoDB 支持事务、行锁、MVCC、崩溃恢复；MyISAM 不支持事务，主要是表锁。

4. **一行记录怎么存储？**  
   InnoDB 页内存储行记录，记录包含变长字段长度、NULL 位图、记录头、隐藏字段和真实字段。

5. **InnoDB 页默认多大？**  
   默认 16KB。

6. **什么是聚簇索引？**  
   叶子节点存整行数据，InnoDB 主键索引就是聚簇索引。

7. **没有主键 InnoDB 怎么办？**  
   优先选择非空唯一索引，否则生成隐藏 `row_id`。

8. **为什么建议自增主键？**  
   插入顺序递增，减少页分裂和随机写，二级索引也更小。

9. **UUID 做主键有什么问题？**  
   离散写入导致页分裂，索引更大，缓存命中率更低。

10. **char 和 varchar 区别？**  
    `char` 定长，适合固定长度；`varchar` 变长，节省空间但有长度元信息。

#### 索引

11. **为什么用 B+Tree？**  
    层高低、IO 少，叶子节点链表适合范围查询。

12. **BTree 和 B+Tree 区别？**  
    B+Tree 非叶子节点只存 key，数据在叶子节点，范围扫描更友好。

13. **Hash 索引为什么不适合范围查询？**  
    Hash 只适合等值匹配，不保留有序性。

14. **什么是回表？**  
    二级索引查到主键后，再回聚簇索引查整行。

15. **什么是覆盖索引？**  
    查询字段都在索引里，不需要回表。

16. **联合索引最左前缀是什么？**  
    索引按字段顺序排序，查询必须从最左字段连续使用才能充分利用。

17. **索引下推是什么？**  
    存储引擎在索引遍历时先过滤部分条件，减少回表。

18. **哪些情况索引会失效？**  
    左模糊、函数计算、隐式类型转换、跳过联合索引左列、低选择性字段滥用索引。

19. **`like '%abc'` 为什么不能走普通索引？**  
    前缀不确定，B+Tree 无法定位起始范围。

20. **联合索引字段顺序怎么定？**  
    等值高频字段靠前，范围字段靠后，同时兼顾排序、分组、覆盖索引。

21. **索引越多越好吗？**  
    不是。索引增加写入成本、占空间，并可能干扰优化器选择。

22. **低基数字段适合建索引吗？**  
    单独建通常收益低，和其他字段组合可能有价值。

23. **`order by` 如何利用索引？**  
    排序字段顺序和方向与索引一致，且 where 条件满足索引前缀。

24. **`group by` 慢怎么优化？**  
    建合适索引、减少分组数据量、预聚合、异步统计。

25. **`count(*)`、`count(1)`、`count(col)` 区别？**  
    InnoDB 中 `count(*)` 通常被优化；`count(col)` 不统计 NULL。

#### EXPLAIN 与 SQL 优化

26. **EXPLAIN 看哪些字段？**  
    `type`、`key`、`rows`、`filtered`、`Extra`。

27. **type 从好到差？**  
    `system`、`const`、`eq_ref`、`ref`、`range`、`index`、`ALL`。

28. **`Using filesort` 一定慢吗？**  
    不一定，但表示不能直接用索引顺序完成排序，大数据量要关注。

29. **`Using temporary` 说明什么？**  
    使用临时表，常见于排序、分组、去重。

30. **深分页怎么优化？**  
    使用游标分页、延迟关联、记录上次最大 id，避免大 offset 扫描。

31. **大表加字段怎么做？**  
    评估是否 Online DDL，低峰执行，必要时用影子表或 gh-ost/pt-osc。

32. **大批量更新怎么做？**  
    分批、按主键范围、控制事务大小、避免长事务和主从延迟。

33. **为什么不要 `select *`？**  
    增加 IO、网络传输、回表概率，也不利于覆盖索引。

34. **OR 条件怎么优化？**  
    确认两侧都有索引，必要时改写为 `UNION ALL`。

35. **隐式类型转换有什么风险？**  
    可能导致索引失效，并产生非预期匹配。

#### 事务与 MVCC

36. **ACID 是什么？**  
    原子性、一致性、隔离性、持久性。

37. **四种隔离级别？**  
    读未提交、读已提交、可重复读、串行化。

38. **InnoDB 默认隔离级别？**  
    可重复读。

39. **MVCC 解决什么问题？**  
    通过版本链和 ReadView 支持一致性非锁定读，提高读写并发。

40. **undo log 作用？**  
    支持事务回滚和 MVCC 历史版本读取。

41. **redo log 作用？**  
    保证崩溃恢复，先写日志再刷数据页。

42. **binlog 作用？**  
    Server 层日志，用于主从复制和数据恢复。

43. **redo log 和 binlog 区别？**  
    redo 是 InnoDB 物理日志，循环写；binlog 是 Server 层逻辑日志，追加写。

44. **两阶段提交解决什么？**  
    保证 redo log 和 binlog 状态一致，避免崩溃后主从或恢复数据不一致。

45. **什么是当前读和快照读？**  
    当前读读最新并加锁；快照读基于 ReadView 读历史版本。

46. **可重复读如何避免幻读？**  
    快照读靠 MVCC，当前读靠 next-key lock。

47. **长事务有什么危害？**  
    持有锁、阻塞 purge、undo 膨胀、主从延迟、影响 DDL。

48. **死锁怎么排查？**  
    看 `SHOW ENGINE INNODB STATUS`，统一加锁顺序，缩短事务，命中索引。

49. **间隙锁是什么？**  
    锁住索引记录之间的范围，防止其他事务插入导致幻读。

50. **什么情况下行锁变表锁？**  
    InnoDB 仍是索引记录锁，但如果条件不走索引会扫描大量记录并加锁，效果类似表锁。

#### 主从、分库分表与高可用

51. **MySQL 主从复制流程？**  
    主库写 binlog，从库 IO 线程拉取 relay log，SQL 线程重放。

52. **主从延迟原因？**  
    大事务、从库性能弱、SQL 单线程瓶颈、网络延迟、从库查询压力大。

53. **如何降低主从延迟？**  
    拆小事务、并行复制、从库扩容、读写分离策略、避免从库重查询。

54. **读写分离有什么坑？**  
    写后马上读可能读到旧数据，需要读主库、延迟检测或业务容忍。

55. **分库分表按什么拆？**  
    按业务边界和高频查询键拆，常用用户 id、商户 id、订单 id。

56. **分库分表后难点？**  
    跨库 Join、分布式事务、全局唯一 id、分页排序、聚合统计、扩容迁移。

57. **全局 ID 怎么设计？**  
    雪花算法、号段模式、数据库序列、Redis 自增，各有可用性和趋势递增取舍。

58. **冷热数据怎么分离？**  
    热数据保留在线库，冷数据归档到历史库或数仓，查询入口区分。

59. **数据库连接池怎么调？**  
    结合 MySQL 最大连接数、应用实例数、单机并发和慢请求情况设置。

60. **MySQL CPU 高怎么排查？**  
    查慢 SQL、全表扫描、排序分组、锁等待、连接数、执行计划变化。

#### 场景题

61. **订单表亿级怎么查？**  
    按商户/用户/时间分片，建立复合索引，冷热归档，避免任意条件全局查。

62. **库存扣减如何防超卖？**  
    `update stock set num=num-1 where sku_id=? and num>0` 原子扣减，或 Redis 预扣加异步落库。

63. **余额扣减怎么保证安全？**  
    数据库事务、条件更新、流水唯一号、幂等、对账。

64. **唯一约束有什么价值？**  
    数据库层兜底防重复，是幂等和并发安全的重要手段。

65. **如何做数据恢复？**  
    全量备份 + binlog 增量恢复，按时间点回放到故障前。

66. **如何防 SQL 注入？**  
    预处理绑定参数、ORM 参数化、白名单排序字段，不拼接用户输入。

67. **如何处理表结构变更风险？**  
    兼容发布，先加字段后写代码，双写校验，再切读，最后清理旧字段。

68. **如何分析锁等待？**  
    查 `information_schema.innodb_trx`、`performance_schema`、慢日志和事务 SQL。

69. **为什么索引命中但还是慢？**  
    回表多、扫描范围大、数据不在缓存、排序临时表、锁等待、返回数据太多。

70. **如何设计审计日志表？**  
    只追加不修改，按业务 id 和时间建索引，冷热归档，敏感字段脱敏。

71. **如何优化模糊搜索？**  
    前缀搜索用索引，全文检索用 ES/OpenSearch，短文本可用倒排或冗余字段。

72. **如何避免 N+1 查询？**  
    批量查询、预加载、Join 或手动组装 map。

73. **大字段怎么处理？**  
    拆表或对象存储，只在需要时加载，主表保留核心查询字段。

74. **统计报表怎么做？**  
    业务库不跑重聚合，使用异步汇总、宽表、数仓或缓存。

75. **为什么要设置 SQL 超时？**  
    防止慢 SQL 长时间占用连接和 worker，造成级联故障。

76. **MySQL 磁盘 IO 高怎么办？**  
    看 buffer pool 命中率、慢 SQL、临时表落盘、redo/binlog 刷盘策略。

77. **buffer pool 是什么？**  
    InnoDB 缓存数据页和索引页的核心内存区域。

78. **change buffer 是什么？**  
    缓存非唯一二级索引变更，减少随机 IO。

79. **自适应哈希索引是什么？**  
    InnoDB 根据热点访问自动建立内存哈希索引，加速等值查询。

80. **Online DDL 一定无锁吗？**  
    不是。不同操作和版本锁粒度不同，仍可能短暂阻塞读写。

81. **为什么删除大量数据后磁盘不变小？**  
    InnoDB 表空间可能只标记空闲页，不自动归还 OS，需要重建表。

82. **如何做唯一业务流水？**  
    业务单号唯一索引 + 状态机 + 幂等表。

83. **乐观锁怎么实现？**  
    版本号或更新时间条件更新，更新失败说明并发冲突。

84. **悲观锁怎么实现？**  
    事务内 `select ... for update` 锁定记录。

85. **`for update` 注意什么？**  
    必须在事务内，条件要命中索引，避免锁范围扩大。

86. **为什么要避免长连接空闲过多？**  
    占用 MySQL 连接资源，可能导致新请求无法连接。

87. **如何做多租户数据隔离？**  
    所有表带 tenant_id 并建组合索引，关键查询强制带租户条件。

88. **如何保证迁移数据正确？**  
    双写、增量同步、校验总量和抽样明细、灰度切读、可回滚。

89. **SQL 优化最怕什么？**  
    只凭感觉加索引，不看执行计划和数据分布。

90. **如何设计支付流水表？**  
    流水号唯一、状态机、金额不可变、请求响应留痕、对账字段、时间索引。

91. **如何防止重复下单？**  
    用户维度幂等 key、订单唯一约束、Redis 短锁、状态机兜底。

92. **联合索引 `(a,b,c)` 查询 `a=? and c=?` 怎么用？**  
    只能有效利用 `a` 定位，`c` 可能用于索引下推过滤，不能利用 `b,c` 的有序性。

93. **范围条件后面的索引还能用吗？**  
    对排序和定位通常不能继续充分利用，但可能用于索引下推。

94. **为什么 `not in` 可能慢？**  
    选择性差且 NULL 语义复杂，可能不易优化，可改写为 anti join 或业务拆分。

95. **如何处理热点行更新？**  
    分桶、异步汇总、Redis 预聚合、降低同一行写竞争。

96. **如何设计可扩展订单状态？**  
    状态机表述允许流转，禁止随意更新，记录状态流水。

97. **数据库如何配合缓存一致性？**  
    先写库后删缓存，缓存未命中再回源，必要时延迟双删或订阅 binlog。

98. **为什么不建议缓存和数据库双写？**  
    两个写操作无法天然原子，失败时容易不一致。

99. **如何防止 DDL 影响线上？**  
    低峰、灰度、Online DDL 工具、观察锁等待和主从延迟。

100. **如何回答“你怎么做 MySQL 调优”？**  
     从监控、慢日志、执行计划、索引、SQL 改写、架构拆分、验证效果完整闭环回答。

101. **什么是 MDL 元数据锁？**  
     访问表结构时持有的锁，长事务可能阻塞 DDL，DDL 又阻塞后续查询。

102. **如何发现执行计划抖动？**  
     比较慢 SQL 样本、统计信息、索引选择，必要时分析数据分布和优化器估算。

103. **如何设计高并发插入表索引？**  
     控制索引数量，主键趋势递增，批量写入，必要时分表。

104. **MySQL 和 Redis 分布式锁怎么选？**  
     MySQL 适合强一致低频锁，Redis 适合高频短锁但必须设计过期、唯一值、续期和幂等。

105. **事务里能调用外部接口吗？**  
     尽量不要。外部接口慢会延长锁持有，建议事务内只改本地状态，事务后发事件异步处理。

## 四、Redis 从数据结构到高并发

### 1. 核心机制

Redis 快的原因：

- 数据主要在内存中。
- 单线程执行命令，避免复杂锁竞争。
- IO 多路复用处理大量连接。
- 数据结构针对场景优化。
- Redis 6 之后网络 IO 可多线程，命令执行仍以单线程为核心。

### 2. 数据结构与场景

- String：缓存、计数器、分布式锁、验证码。
- Hash：对象缓存、用户属性、购物车。
- List：简单队列、时间线，但复杂消息建议 Stream 或 MQ。
- Set：去重、标签、共同好友。
- ZSet：排行榜、延迟队列、权重排序。
- Bitmap：签到、活跃状态。
- HyperLogLog：UV 估算。
- Stream：消费组、轻量消息流。

### 3. 高并发缓存问题

- 穿透：查不存在数据。解决：缓存空值、布隆过滤器、参数校验。
- 击穿：热点 key 过期。解决：互斥锁、逻辑过期、热点不过期。
- 雪崩：大量 key 同时过期或 Redis 故障。解决：过期随机值、多级缓存、限流降级、集群高可用。
- 大 key：拆分、压缩、分页读取、异步清理。
- 热 key：本地缓存、多副本、读写分离、热点发现、限流。

### 4. Redis 80+ 高频面试题

1. **Redis 为什么快？**  
   内存存储、单线程命令执行、IO 多路复用、高效数据结构。

2. **Redis 单线程为什么还能高并发？**  
   命令执行快且避免锁竞争，网络层通过事件循环处理大量连接。

3. **Redis 6 多线程做了什么？**  
   主要用于网络 IO 读写和协议解析，命令执行仍基本单线程。

4. **String 底层是什么？**  
   SDS，支持动态扩容、二进制安全、O(1) 获取长度。

5. **Hash 适合什么？**  
   存对象字段，便于局部更新，但字段过多会变大 key。

6. **List 使用场景？**  
   简单队列、最新列表；复杂可靠消息不要只靠 List。

7. **Set 使用场景？**  
   去重、集合运算、标签关系。

8. **ZSet 使用场景？**  
   排行榜、延迟任务、按分数范围查询。

9. **Bitmap 使用场景？**  
   用户签到、活跃统计、布尔状态集合。

10. **HyperLogLog 缺点？**  
    只能估算基数，有误差，不能取明细。

11. **Stream 比 List 强在哪？**  
    支持消息 ID、消费组、ACK、pending 列表，更像轻量消息队列。

12. **缓存穿透是什么？**  
    请求不存在数据，缓存没有，数据库也没有。

13. **穿透怎么解决？**  
    参数校验、缓存空值、布隆过滤器。

14. **缓存击穿是什么？**  
    热点 key 过期，大量请求同时打到数据库。

15. **击穿怎么解决？**  
    互斥锁、逻辑过期、热点 key 不过期。

16. **缓存雪崩是什么？**  
    大量 key 同时失效或 Redis 不可用，压力涌向数据库。

17. **雪崩怎么解决？**  
    TTL 加随机值、高可用、限流降级、多级缓存。

18. **缓存和数据库如何保持一致？**  
    常用先写数据库再删缓存，失败重试或延迟补偿。

19. **为什么不是更新缓存？**  
    更新缓存可能写入脏值或浪费资源，删除后懒加载更稳。

20. **延迟双删是什么？**  
    写库后删缓存，过一段时间再删一次，降低并发读写导致脏缓存概率。

21. **分布式锁怎么实现？**  
    `SET key value NX EX seconds`，value 唯一，释放时 Lua 校验 value 后删除。

22. **为什么释放锁要 Lua？**  
    校验和删除必须原子，避免删掉别人的锁。

23. **锁过期但业务没执行完怎么办？**  
    续期、缩短业务、幂等补偿，避免只依赖锁保证正确性。

24. **RedLock 了解吗？**  
    多 Redis 节点多数派加锁方案，但复杂度高，强一致场景更建议数据库或业务幂等兜底。

25. **Redis 持久化方式？**  
    RDB 快照、AOF 追加日志、混合持久化。

26. **RDB 优缺点？**  
    恢复快、文件紧凑，但两次快照之间可能丢数据。

27. **AOF 优缺点？**  
    数据更安全，可按秒刷盘，但文件更大，恢复较慢。

28. **AOF 重写是什么？**  
    压缩历史命令，生成能恢复当前数据集的最小命令集合。

29. **Redis 过期删除策略？**  
    惰性删除 + 定期删除。

30. **内存淘汰策略有哪些？**  
    noeviction、volatile-lru、allkeys-lru、volatile-ttl、lfu 等。

31. **LRU 和 LFU 区别？**  
    LRU 看最近使用，LFU 看使用频率。

32. **大 key 有什么危害？**  
    阻塞事件循环、网络传输大、删除慢、集群迁移慢。

33. **如何发现大 key？**  
    `redis-cli --bigkeys`、scan 抽样、监控内存和命令耗时。

34. **如何删除大 key？**  
    用 `UNLINK` 异步释放，或分批删除集合元素。

35. **热 key 有什么危害？**  
    单节点压力过高，导致延迟升高。

36. **如何解决热 key？**  
    本地缓存、读副本、key 分片、限流、提前识别热点。

37. **Redis Cluster 如何分片？**  
    16384 个 hash slot，key 映射到 slot，slot 分配到节点。

38. **Cluster 跨 slot 多 key 操作怎么办？**  
    使用 hash tag 保证同槽，或避免跨 key 原子操作。

39. **主从复制流程？**  
    从库同步主库数据，首次全量同步，之后增量复制。

40. **哨兵作用？**  
    监控主从、故障转移、通知客户端新主节点。

41. **Redis 脑裂风险？**  
    网络分区导致旧主仍可写，恢复后数据可能丢失，需要合理配置最小从库和延迟。

42. **pipeline 有什么用？**  
    批量发送命令，减少网络 RTT。

43. **事务 MULTI/EXEC 能回滚吗？**  
    不像数据库事务，运行时错误不会整体回滚。

44. **Lua 脚本作用？**  
    保证多步 Redis 操作原子执行。

45. **慢查询怎么查？**  
    `SLOWLOG GET`，关注慢命令和大 key 操作。

46. **`KEYS *` 为什么危险？**  
    阻塞 Redis，生产使用 `SCAN` 渐进遍历。

47. **`SCAN` 会重复吗？**  
    可能重复，业务要能去重。

48. **计数器怎么做？**  
    `INCR`/`INCRBY`，必要时设置过期时间。

49. **限流怎么做？**  
    固定窗口、滑动窗口、令牌桶，Redis 可用 Lua 保证原子。

50. **排行榜怎么做？**  
    ZSet 分数排序，`ZINCRBY` 更新分数，`ZREVRANGE` 查询排名。

51. **延迟队列怎么做？**  
    ZSet score 存执行时间，消费者轮询到期任务；高可靠建议 MQ。

52. **购物车用什么结构？**  
    Hash，field 为 sku，value 为数量或 JSON。

53. **签到怎么做？**  
    Bitmap 按用户和月份记录每日签到位。

54. **UV 统计怎么做？**  
    HyperLogLog，接受误差时使用。

55. **布隆过滤器原理？**  
    多个 hash 映射位数组，判断不存在一定不存在，存在可能误判。

56. **Redis 连接数打满怎么办？**  
    查连接泄漏、慢命令、连接池配置、客户端超时和服务端 maxclients。

57. **Redis CPU 高怎么办？**  
    查慢命令、大 key、热 key、Lua 脚本、持久化和网络流量。

58. **Redis 内存满怎么办？**  
    分析 key 分布，清理过期无效数据，调整淘汰策略，扩容或拆分。

59. **缓存预热是什么？**  
    发布或活动前提前加载热点数据，避免冷启动打爆数据库。

60. **缓存降级是什么？**  
    Redis 异常时返回默认值、本地缓存或限制非核心功能。

61. **本地缓存和 Redis 怎么配合？**  
    本地缓存抗热点，Redis 做共享缓存；注意过期和更新通知。

62. **为什么 Redis 不适合存超大对象？**  
    占内存、传输慢、阻塞操作，且不利于局部更新。

63. **如何设计库存缓存？**  
    Redis 预扣减，扣减成功进入 MQ，DB 最终确认；失败要回补或冻结库存。

64. **如何保证 Redis 预扣不超卖？**  
    Lua 原子判断库存大于 0 再扣减。

65. **如何防重复领取优惠券？**  
    用户券唯一键、Redis `SETNX`、数据库唯一索引兜底。

66. **Redis 做队列有什么问题？**  
    可靠性、ACK、重试、堆积观测不如专业 MQ。

67. **Stream pending 是什么？**  
    已投递未 ACK 的消息列表，可用于重投和故障恢复。

68. **Redis 和 Memcached 区别？**  
    Redis 数据结构更丰富，支持持久化、复制、Lua、集群能力。

69. **缓存 key 如何设计？**  
    业务前缀、版本、主键、租户维度，避免过长和歧义。

70. **TTL 如何设计？**  
    根据数据实时性设置，热点加随机值，核心配置可主动失效。

71. **如何做缓存版本控制？**  
    key 加版本号或命名空间，发布时切换版本避免批量删除。

72. **如何避免缓存击穿时锁失效？**  
    锁内二次检查缓存，锁超时合理，失败请求短暂等待或返回旧值。

73. **逻辑过期是什么？**  
    value 内存过期时间，过期后先返回旧值并异步刷新。

74. **Redis 延迟高怎么定位？**  
    latency monitor、slowlog、网络、持久化、fork、swap、大 key。

75. **fork 对 Redis 有什么影响？**  
    RDB/AOF 重写时 fork 子进程，数据大时可能造成延迟抖动。

76. **为什么禁止 swap？**  
    内存换页会让 Redis 延迟急剧上升。

77. **Redis 安全怎么做？**  
    内网访问、密码、ACL、禁用危险命令、最小权限。

78. **如何做多环境隔离？**  
    不同实例或 db/key 前缀，生产不要随意使用 `FLUSHDB`。

79. **Redis 作为配置中心合适吗？**  
    可做简单动态配置，但需要持久化、审计、推送和回滚能力时应使用专门配置中心。

80. **Redis 面试压轴怎么答？**  
    从数据结构、线程模型、持久化、高可用、缓存一致性、热点问题、线上排障形成闭环。

81. **Redis 缓存命中率低怎么办？**  
    分析 key、TTL、访问模式，调整粒度，预热热点，避免无效缓存。

82. **如何避免缓存污染？**  
    对一次性冷数据不缓存或短 TTL，使用 LFU 策略。

83. **什么场景不用 Redis？**  
    强一致事务、超大文件、复杂关系查询、必须永久可靠消息。

## 五、RabbitMQ 与消息架构

### 1. 核心链路

Producer 发送消息到 Exchange，Exchange 根据类型和 routing key 路由到 Queue，Consumer 消费后 ACK。可靠性要看三段：生产者到 Broker、Broker 持久化、Broker 到消费者。

### 2. 可靠投递

- 生产者：Publisher Confirm，失败重试或落库。
- 路由：mandatory + return callback，避免消息到 Exchange 后无队列接收。
- Broker：durable exchange、durable queue、persistent message。
- 消费者：手动 ACK，失败重试，超过次数进死信队列。
- 业务：幂等表、唯一索引、消息状态表。

### 3. 高频题

1. **MQ 解决什么问题？**  
   解耦、异步、削峰、最终一致性。

2. **MQ 会带来什么问题？**  
   消息丢失、重复消费、顺序问题、延迟、堆积、运维复杂度。

3. **如何保证消息不丢？**  
   confirm、持久化、手动 ACK、失败重试、死信和补偿。

4. **如何处理重复消费？**  
   消息唯一 ID、业务唯一索引、消费记录表、幂等状态机。

5. **如何保证顺序？**  
   同一业务键进同一队列，单消费者处理；代价是吞吐下降。

6. **消息堆积怎么办？**  
   先保护系统，再扩消费者、优化消费逻辑、临时扩队列、限流生产者，最后补偿。

7. **死信队列用来做什么？**  
   承接拒绝、过期、超过长度的消息，用于重试、排查和人工补偿。

8. **延迟队列怎么做？**  
   TTL + DLX 或延迟插件；订单超时关闭是典型场景。

9. **prefetch 有什么用？**  
   控制消费者未 ACK 消息数量，避免慢消费者积压太多消息。

10. **事务消息怎么设计？**  
    本地事务写业务表和消息表，事务后异步投递，失败重试，消费者幂等。

## 六、Go 并发与微服务

### 1. Go 面试核心

- goroutine：轻量级并发执行单元。
- channel：协程间通信和同步。
- context：超时、取消、链路传值。
- sync：Mutex、RWMutex、WaitGroup、Once、Pool。
- GMP：Goroutine、Machine、Processor 调度模型。
- GC：并发三色标记，关注低延迟。

### 2. 高频题

1. **goroutine 和线程区别？**  
   goroutine 由 Go runtime 调度，初始栈小，可大量创建；线程由 OS 调度，成本更高。

2. **channel 关闭后读写行为？**  
   读已关闭 channel 会读到零值和 `ok=false`；写已关闭 channel 会 panic。

3. **如何优雅停止 goroutine？**  
   用 context、关闭 channel、WaitGroup 等传递退出信号并等待收尾。

4. **map 并发安全吗？**  
   不安全，并发读写要加锁或用 `sync.Map`。

5. **defer 顺序？**  
   后进先出，函数返回前执行。

6. **nil interface 坑是什么？**  
   接口包含类型和值，只有二者都为空才等于 nil。

7. **GMP 是什么？**  
   G 是 goroutine，M 是系统线程，P 是调度器上下文，P 维护本地队列并把 G 调度到 M 上执行。

8. **context 不能放什么？**  
   不放大对象和可选参数，主要放取消、超时、trace id 等请求级数据。

9. **微服务接口超时怎么设计？**  
   客户端超时、服务端超时、熔断、重试退避、幂等和链路追踪。

10. **Go 服务如何分层？**  
    handler 处理协议，service 处理业务，repository 处理存储，model 定义数据结构。

### 3. 微服务落地关注点

- 服务注册发现：服务实例动态上下线。
- 配置中心：配置分环境、灰度、回滚。
- 网关：鉴权、限流、路由、灰度。
- 链路追踪：trace id 贯穿 HTTP、RPC、MQ。
- 熔断降级：防止下游故障拖垮上游。
- 幂等和补偿：跨服务调用不能只依赖一次成功。

## 七、Docker、Kubernetes、Nginx、Linux 运维

### 1. Docker

高频点：

- 镜像是分层只读文件系统，容器是在镜像上加可写层。
- Dockerfile 要减少层数、利用缓存、避免把密钥写入镜像。
- 多阶段构建可减小镜像体积。
- 容器日志要输出到 stdout/stderr，由平台采集。

常见命令：

```bash
docker build -t app:latest .
docker run -d --name app -p 8080:80 app:latest
docker logs -f app
docker exec -it app sh
```

### 2. Kubernetes

核心对象：

- Pod：最小调度单元。
- Deployment：声明式管理副本和滚动发布。
- Service：稳定访问入口。
- Ingress：HTTP 入口和路由。
- ConfigMap/Secret：配置和敏感信息。
- HPA：自动扩缩容。

面试高频：

- **Pod 一直重启怎么查？**  
  `kubectl describe pod` 看事件，`kubectl logs --previous` 看上次崩溃日志，检查探针、资源限制、配置、依赖服务。

- **滚动发布如何回滚？**  
  Deployment 保留 ReplicaSet 历史，可 `kubectl rollout undo` 回滚。

- **readiness 和 liveness 区别？**  
  readiness 决定是否接流量，liveness 决定是否重启容器。

### 3. Nginx

核心能力：

- 反向代理、负载均衡、静态资源、HTTPS、限流、缓存、日志。

高频配置点：

```nginx
location / {
    proxy_pass http://backend;
    proxy_connect_timeout 3s;
    proxy_read_timeout 30s;
}
```

面试题：

- **Nginx 502 和 504 区别？**  
  502 多为上游不可用或返回异常；504 是上游响应超时。

- **如何限流？**  
  `limit_req_zone` + `limit_req` 按 key 控制请求速率。

- **Nginx 如何转发 PHP？**  
  通过 FastCGI 转发给 PHP-FPM，不直接执行 PHP。

### 4. Linux 排障

常用命令：

```bash
top
free -m
df -h
iostat -x 1
netstat -antp
ss -antp
lsof -i :80
journalctl -u nginx
tail -f app.log
```

排障模板：

- CPU 高：top 定位进程，查看线程和热点日志，确认死循环、慢 SQL、序列化、加密压缩等。
- 内存高：free、ps、容器 limit、应用堆/常驻数组/缓存。
- 磁盘满：df、du，清日志、归档、扩容，避免直接删正在写的文件不释放句柄。
- 连接多：ss/netstat，看 ESTABLISHED、TIME_WAIT、CLOSE_WAIT。
- 端口不通：防火墙、安全组、监听地址、服务状态、DNS。

## 八、系统设计专题

### 1. 积分系统

核心表：

- 用户积分账户表：当前余额、冻结积分、版本号。
- 积分流水表：增加/扣减明细，业务单号唯一。
- 积分规则表：来源、有效期、倍率、上限。

关键设计：

- 积分变更必须有流水，不能只改余额。
- 业务单号唯一，防重复发放。
- 扣减用事务和条件更新，余额不足直接失败。
- 过期积分用定时任务分批处理。
- 对账以流水为准，账户余额可重算校验。

### 2. 库存系统

库存模型：

- 可售库存：用户可购买数量。
- 锁定库存：下单未支付占用数量。
- 实物库存：仓库真实库存。

防超卖方案：

- 低并发：数据库条件更新。
- 高并发：Redis 预扣 + MQ 异步落库 + 失败回补。
- 最终一致：库存流水、补偿任务、定期校验。

状态流：

```text
下单锁定库存 -> 支付成功扣减锁定 -> 超时取消释放锁定
```

### 3. 提现系统

核心风险：

- 重复提现、余额超扣、通道重复回调、风控拦截、对账差异。

设计要点：

- 余额账户和流水强事务。
- 提现单号唯一，状态机控制。
- 提交、审核、打款、成功、失败、退回状态清晰。
- 第三方回调幂等，根据通道流水号去重。
- 每日对账，差异进入人工处理。
- 大额提现走风控和人工审核。

### 4. 消息中心

消息类型：

- 站内信、短信、邮件、Push、企业微信。

设计要点：

- 模板管理：变量、语言、渠道。
- 发送任务：业务方提交消息事件。
- 渠道路由：根据用户偏好和渠道可用性选择。
- 幂等：业务消息 ID + 用户 + 渠道唯一。
- 失败重试：短失败重试，永久失败落库。
- 频控：用户维度、业务维度、渠道维度限流。

### 5. 秒杀系统

核心原则：少查库、早限流、异步化、防超卖、可降级。

链路：

1. 活动配置预热到 Redis。
2. 网关和接口限流。
3. Redis Lua 原子扣库存。
4. 成功请求进入 MQ。
5. 消费者创建订单，DB 唯一约束防重复。
6. 超时未支付释放库存。

### 6. 订单系统

设计重点：

- 订单主表、订单明细、支付单、状态流水。
- 订单号全局唯一且趋势递增。
- 状态机禁止非法流转。
- 支付回调幂等。
- 订单查询按用户、商户、时间建立索引。
- 大表按时间或业务键分库分表，历史订单归档。

## 九、高并发与分布式架构

### 1. 高并发三板斧

- 缓存：减少数据库压力。
- 队列：削峰填谷，异步处理。
- 限流降级：保护核心链路。

### 2. 分布式一致性

常见方案：

- 本地事务：单库内强一致。
- TCC：Try、Confirm、Cancel，适合强业务控制但开发成本高。
- Saga：长事务拆成多个本地事务，失败走补偿。
- 可靠消息最终一致：本地消息表 + MQ + 消费幂等。

回答原则：

强一致成本高，不是所有业务都需要。支付、余额、提现更偏强一致；通知、积分、报表可接受最终一致。

### 3. 幂等设计

幂等不是“防止调用多次”，而是“调用多次结果一致”。

常用方案：

- 请求唯一号。
- 数据库唯一索引。
- 状态机判断。
- Redis `SETNX` 短期防抖。
- 消费记录表。

### 4. 限流、熔断、降级

- 限流：控制入口流量，如令牌桶、漏桶、固定窗口。
- 熔断：下游持续失败时短时间不再调用。
- 降级：返回默认值、旧数据、关闭非核心功能。

### 5. 线上故障复盘模板

1. 故障现象：时间、影响范围、错误率、延迟、用户影响。
2. 直接原因：慢 SQL、缓存故障、消息堆积、发布变更等。
3. 根因：容量评估不足、缺少限流、SQL 无索引、监控缺失。
4. 处理过程：止血、回滚、扩容、修复、补偿数据。
5. 防复发：监控告警、压测、灰度、变更检查、演练。

## 十、项目深度拆解

### 1. 项目讲解通用结构

```text
这个项目解决什么业务问题？
我的职责是什么？
核心链路是什么？
技术难点在哪里？
我做了哪些设计？
上线后数据如何？
出过什么问题，怎么解决？
如果重做会怎么优化？
```

### 2. DCT 项目讲法

可从“数据采集、业务流转、权限、报表、异步任务”展开。

亮点准备：

- 多角色权限：角色、菜单、按钮、数据权限。
- 流程状态机：避免状态乱跳。
- 报表查询优化：宽表、异步统计、缓存。
- 文件导入导出：分批、队列、失败明细。
- 审计日志：关键操作可追溯。

### 3. WMS 仓储系统

核心业务：

- 入库、上架、拣货、复核、出库、盘点、库存调整。

面试重点：

- 库存不能只看一个数字，要区分可用、锁定、在途、冻结。
- 每次库存变化必须有库存流水。
- 并发扣减要用条件更新或锁。
- 盘点差异需要审批和调整记录。
- 和 TMS、订单系统通过事件或接口解耦。

### 4. TMS 运输系统

核心业务：

- 运输单、承运商、线路、车辆、司机、轨迹、签收、运费结算。

亮点：

- 状态机：待派车、运输中、已签收、异常、取消。
- 轨迹数据量大，冷热分离或时序存储。
- 运费规则复杂，适合策略模式。
- 异常件通知通过 MQ 异步推送。

### 5. 太古中台类项目

讲法重点：

- 中台不是简单公共接口，而是沉淀通用能力和统一数据口径。
- 能力包括用户、权限、商品、订单、库存、消息、报表、审批。
- 对外提供标准 API、事件、SDK 或管理后台。
- 难点是业务差异抽象、权限隔离、数据一致性、版本兼容。

### 6. 项目追问准备

- **你做过最复杂的接口是什么？**  
  选库存、提现、支付回调、消息中心这类有并发、幂等、一致性的问题。

- **你如何证明优化有效？**  
  说优化前后指标：耗时、QPS、错误率、CPU、慢 SQL 数量。

- **你遇到过线上事故吗？**  
  讲事实，不甩锅。说明止血、根因、补偿、防复发。

- **你项目里最有技术含量的点？**  
  不要只说用了 Redis/MQ，要说为什么用、怎么保证正确、出问题怎么兜底。

## 十一、PMP 与技术管理

### 1. 项目管理五大过程组

- 启动：明确目标、干系人、范围边界。
- 规划：拆任务、排期、风险、资源、沟通计划。
- 执行：推进研发、协调资源、解决阻塞。
- 监控：进度、质量、风险、变更。
- 收尾：验收、复盘、文档、经验沉淀。

### 2. 技术负责人关注点

- 目标清晰：需求价值、验收标准、优先级。
- 架构可控：不为炫技引入复杂组件。
- 质量保障：Code Review、测试、灰度、回滚。
- 风险管理：识别延期、依赖、性能、数据风险。
- 团队协作：接口契约、文档、节奏、反馈。

### 3. 面试管理题回答模板

**需求频繁变更怎么办？**

回答：

```text
先确认变更是否影响目标和验收标准，再评估范围、排期和风险。
小变更进入当前迭代，大变更走变更评审，明确取舍：
要么延期，要么砍低优先级需求，要么加资源。
同时保留变更记录，避免口头需求导致责任不清。
```

**技术债怎么处理？**

回答：

```text
技术债要分级处理。影响稳定性、安全、交付效率的优先治理；
纯代码洁癖不单独立大项目。可以在业务迭代中顺手治理，
也可以对高风险模块安排专项重构，但必须有回归测试和灰度方案。
```

**如何带新人？**

回答：

```text
我会给新人明确模块边界、开发规范、环境文档和样例任务。
初期任务切小，配合 Code Review 反馈问题。
逐步让他负责完整小需求，包括设计、开发、测试、上线和复盘。
```

1. 如何评估项目排期？
2. 需求不清楚时如何推进？
3. 线上事故如何组织处理？

8. 如何制定技术规范？

10. 如何向非技术人员解释技术风险？

### 4. 临场回答原则
