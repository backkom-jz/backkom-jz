# 系统设计、项目深拆与技术管理面试手册

## 一、系统设计回答框架

面对系统设计题，先问约束，再给方案：

- QPS、峰值、数据量、增长速度。
- 一致性要求。
- 延迟要求。
- 是否允许异步。
- 失败成本。
- 是否多租户、多区域、多渠道。

回答结构：

1. 核心业务流程。
2. 数据模型。
3. 并发与一致性。
4. 缓存与队列。
5. 异常、重试、补偿。
6. 监控、告警、灰度、回滚。

## 二、积分系统

核心表：

- `point_account`：用户积分余额、冻结积分、版本号。
- `point_flow`：积分流水，业务单号唯一。
- `point_rule`：积分规则。
- `point_expire_plan`：过期计划。

设计原则：

- 任何积分变化必须有流水。
- 业务单号唯一，防重复发放。
- 扣减使用事务和条件更新。
- 过期积分分批处理。
- 余额可由流水重算校验。

高频追问：

- **重复发积分怎么办？** 业务单号唯一索引兜底。
- **积分扣成负数怎么办？** 条件更新 `balance >= amount`。
- **积分过期怎么做？** 定时任务扫描过期计划，分批扣减并写流水。

## 三、库存系统

库存字段：

- 可售库存。
- 锁定库存。
- 已售库存。
- 冻结库存。

流程：

```text
下单 -> 锁定库存 -> 支付成功扣减锁定 -> 超时取消释放锁定
```

高并发方案：

- 低并发：数据库条件更新。
- 高并发：Redis Lua 预扣，MQ 异步落库。
- 强一致库存：数据库事务和库存流水。

防超卖关键：

- 原子扣减。
- 库存流水。
- 订单唯一约束。
- 失败回补。
- 定期对账。

## 四、提现系统

核心表：

- 账户表。
- 账户流水表。
- 提现单表。
- 通道请求表。
- 对账差异表。

状态流：

```text
提交 -> 风控中 -> 待审核 -> 待打款 -> 打款中 -> 成功/失败/退回
```

关键点：

- 余额扣减和提现单创建在同一事务。
- 提现单号唯一。
- 第三方回调幂等。
- 失败要退回余额并写反向流水。
- 每日对账处理差异。
- 大额提现走审核和风控。

## 五、消息中心

能力模块：

- 模板管理。
- 消息任务。
- 渠道路由。
- 用户订阅偏好。
- 发送记录。
- 失败重试。
- 频控限流。

设计亮点：

- 业务方只发消息事件，不关心具体渠道。
- 模板变量校验，避免线上缺变量。
- 短信、邮件、站内信、Push 统一抽象为 Channel。
- 业务消息 ID + 用户 + 渠道唯一，防重复发送。
- MQ 异步发送，失败按次数重试，最终落失败表。

## 六、高并发架构

高并发不是只加机器，核心是减少同步链路压力：

- 静态化：页面和配置尽量静态化。
- 缓存：热点数据放 Redis 或本地缓存。
- 限流：入口限流保护后端。
- 队列：非实时任务异步处理。
- 分片：数据库和缓存按业务键拆。
- 降级：非核心功能可关闭。

## 七、分布式一致性

常见方案：

- 本地事务：单库强一致。
- TCC：强业务控制，开发成本高。
- Saga：长事务拆分，本地事务加补偿。
- 可靠消息最终一致：本地消息表 + MQ + 幂等消费。

选择原则：

- 支付、余额、提现：偏强一致，数据库事务和对账兜底。
- 通知、积分、报表：可最终一致。
- 库存：核心扣减要防超卖，链路可异步。

## 八、项目深度拆解

### 1. DCT 项目

可讲点：

- 多角色权限。
- 流程审批。
- 报表统计。
- 导入导出。
- 操作审计。
- 异步任务。

回答模板：

```text
DCT 项目主要复杂点在流程状态、权限和报表性能。
我负责后端核心模块，做了 Controller/Service/Repository 分层，
流程用状态机控制，关键操作写审计日志，报表用异步汇总表降低主库压力。
```

### 2. WMS 项目

可讲点：

- 入库、上架、拣货、复核、出库、盘点。
- 库存可用、锁定、冻结、在途拆分。
- 库存流水。
- 并发扣减。
- 与订单/TMS 协同。

### 3. TMS 项目

可讲点：

- 运输单状态机。
- 承运商和线路。
- 轨迹数据。
- 异常件通知。
- 运费规则策略模式。
- 签收回传。

### 4. 太古中台类项目

可讲点：

- 用户、权限、商品、订单、库存、消息、报表能力沉淀。
- 统一 API 和事件。
- 多业务线接入。
- 权限隔离和租户隔离。
- 版本兼容。

## 九、PMP 与技术管理

### 1. 五大过程组

- 启动：明确目标、干系人和边界。
- 规划：范围、排期、成本、质量、风险、沟通。
- 执行：资源协调、任务推进。
- 监控：进度、质量、风险、变更。
- 收尾：验收、复盘、文档沉淀。

### 2. 需求变更

回答：

```text
先确认变更是否影响目标和验收，再评估范围、排期和风险。
小变更进入当前迭代，大变更走变更评审。
如果必须做，就明确取舍：延期、砍范围或增加资源。
```

### 3. 技术债

回答：

```text
技术债要分级。影响稳定性、安全和交付效率的优先治理；
纯代码洁癖不单独立项。重构要配合测试、灰度和回滚。
```

### 4. Code Review

关注：

- 业务逻辑是否正确。
- 并发和幂等是否处理。
- SQL 和索引是否合理。
- 错误处理和日志是否充分。
- 是否影响兼容性。
- 是否有测试或验证方式。
