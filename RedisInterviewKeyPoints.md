# Redis 从数据结构到高并发面试手册

## 一、核心定位

Redis 在后端系统中主要承担缓存、计数、限流、排行榜、分布式锁、轻量队列、热点数据加速等角色。面试回答时不要只说“Redis 很快”，要能讲清楚它为什么快、什么场景适合、什么场景不适合、线上出问题怎么排查。

Redis 快的主要原因：

- 基于内存读写。
- 单线程执行命令，避免锁竞争和频繁上下文切换。
- IO 多路复用处理大量连接。
- SDS、跳表、哈希表、整数集合、压缩列表/listpack 等高效数据结构。
- Redis 6 之后网络 IO 可多线程，核心命令执行仍以单线程模型为主。

## 二、数据结构与典型场景

| 数据结构 | 典型场景 | 面试提醒 |
| --- | --- | --- |
| String | 缓存、计数器、验证码、锁 | 注意 value 大小和过期时间 |
| Hash | 对象缓存、购物车、用户属性 | 字段过多会形成大 key |
| List | 简单队列、最新列表 | 可靠消息优先用 MQ 或 Stream |
| Set | 去重、标签、共同好友 | 适合集合运算 |
| ZSet | 排行榜、延迟队列 | score 可存分数或时间戳 |
| Bitmap | 签到、活跃状态 | 节省布尔状态存储 |
| HyperLogLog | UV 估算 | 有误差，不能取明细 |
| Stream | 消费组、轻量消息流 | 支持 ACK 和 pending |

## 三、缓存高并发问题

### 1. 缓存穿透

查询的数据不存在，缓存没有，数据库也没有，导致请求一直打到数据库。

解决：

- 参数校验，拦截非法 id。
- 缓存空值，TTL 较短。
- 布隆过滤器判断可能存在性。

### 2. 缓存击穿

热点 key 突然过期，大量并发请求同时回源数据库。

解决：

- 互斥锁，只允许一个请求回源。
- 逻辑过期，先返回旧值，异步刷新。
- 热点 key 不设置物理过期，通过后台刷新。

### 3. 缓存雪崩

大量 key 同时失效，或 Redis 集群不可用，流量打到数据库。

解决：

- TTL 加随机值。
- 多级缓存。
- Redis 主从/哨兵/Cluster。
- 限流、降级、熔断。
- 核心数据预热。

### 4. 缓存一致性

常用策略：先写数据库，再删除缓存。缓存未命中时读库并回填。

为什么不是先删缓存再写库：

- 并发读可能在写库前读到旧数据并回填旧缓存。

为什么不是直接更新缓存：

- 更新缓存和写数据库不是原子操作，失败会不一致；并且很多缓存不一定会被再次访问。

增强方案：

- 删除缓存失败进入重试队列。
- 延迟双删降低并发窗口。
- 订阅 binlog 做最终一致同步。
- 对强一致业务，读取直接走数据库或设计版本号校验。

## 四、分布式锁

基础命令：

```text
SET lock_key unique_value NX EX 10
```

释放锁必须校验 value，建议 Lua 脚本保证原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

注意事项：

- value 必须唯一，不能所有客户端都用同一个值。
- 必须设置过期时间，防止死锁。
- 业务执行可能超过锁过期时间，要考虑续期或幂等补偿。
- 分布式锁不是最终正确性的唯一保障，数据库唯一索引和状态机要兜底。

## 五、持久化与高可用

### 1. RDB

RDB 是快照持久化，把某个时间点的数据集保存到磁盘。

优点：文件紧凑，恢复快。  
缺点：两次快照之间的数据可能丢失，fork 时可能造成延迟抖动。

### 2. AOF

AOF 记录写命令，通过重放命令恢复数据。

优点：数据安全性更高，可配置每秒刷盘。  
缺点：文件更大，恢复相对慢，需要 AOF rewrite。

### 3. 主从、哨兵、Cluster

- 主从：读扩展和数据副本。
- 哨兵：监控、故障转移、通知客户端新主。
- Cluster：通过 16384 个 slot 分片，支持水平扩展。

## 六、线上排障

### 1. Redis 延迟高

排查顺序：

1. `SLOWLOG GET` 查慢命令。
2. 查是否有大 key、热 key。
3. 查 CPU、网络、连接数。
4. 查 RDB/AOF rewrite、fork、磁盘 IO。
5. 查是否发生 swap。

### 2. 内存高

处理方式：

- 找大 key 和高增长 key。
- 检查 TTL 设置。
- 调整淘汰策略。
- 拆分大对象。
- 扩容或迁移热点业务。

### 3. 连接数高

排查：

- 应用连接池是否泄漏。
- 短连接是否过多。
- 慢命令是否导致连接堆积。
- `maxclients` 是否过小。

## 七、80 个高频问题速答

1. Redis 为什么快？内存、单线程命令、IO 多路复用、高效数据结构。
2. Redis 单线程为什么能高并发？命令短小且无锁竞争，事件循环处理连接。
3. Redis 6 多线程解决什么？主要优化网络 IO 和协议解析。
4. String 底层是什么？SDS，二进制安全，O(1) 获取长度。
5. Hash 适合什么？对象字段缓存和购物车。
6. List 适合什么？简单队列和列表，但可靠消息不优先选它。
7. Set 适合什么？去重、标签、共同好友。
8. ZSet 适合什么？排行榜、延迟任务。
9. Bitmap 适合什么？签到、活跃状态。
10. HyperLogLog 适合什么？UV 粗略统计。
11. Stream 解决什么？消费组、ACK、pending 的轻量消息。
12. 缓存穿透是什么？不存在数据持续打数据库。
13. 穿透怎么解决？空值、布隆过滤器、参数校验。
14. 缓存击穿是什么？热点 key 过期导致并发回源。
15. 击穿怎么解决？互斥锁、逻辑过期、热点不过期。
16. 缓存雪崩是什么？大量 key 失效或 Redis 故障。
17. 雪崩怎么解决？TTL 随机、高可用、限流降级。
18. 如何保持缓存一致性？写库后删缓存，失败重试补偿。
19. 为什么释放锁要校验 value？防止删掉别人的锁。
20. 为什么释放锁用 Lua？校验和删除要原子。
21. 锁过期业务没完成怎么办？续期、缩短事务、幂等兜底。
22. RDB 优点？恢复快、文件小。
23. RDB 缺点？可能丢快照后的数据。
24. AOF 优点？数据安全性更高。
25. AOF 缺点？文件大、恢复慢、需要 rewrite。
26. AOF rewrite 是什么？压缩历史命令生成最小恢复日志。
27. 过期删除策略？惰性删除加定期删除。
28. 内存淘汰策略？noeviction、LRU、LFU、TTL 等。
29. LRU 和 LFU 区别？最近使用和使用频率。
30. 大 key 危害？阻塞、传输慢、迁移慢、删除慢。
31. 大 key 怎么删？`UNLINK` 或分批删除。
32. 热 key 危害？单节点压力过高。
33. 热 key 怎么解决？本地缓存、分片、多副本、限流。
34. Cluster 如何分片？16384 个 slot。
35. 跨 slot 多 key 怎么办？hash tag 或避免跨 key 原子操作。
36. Pipeline 作用？减少网络 RTT。
37. MULTI/EXEC 能完全回滚吗？不能像数据库事务那样运行时整体回滚。
38. Lua 适合什么？多命令原子操作。
39. KEYS 为什么危险？全量扫描阻塞 Redis。
40. SCAN 注意什么？可能重复，业务要去重。
41. 计数器怎么做？`INCR`/`INCRBY`。
42. 限流怎么做？窗口、滑动窗口、令牌桶，Lua 保证原子。
43. 排行榜怎么做？ZSet。
44. 延迟队列怎么做？ZSet score 存执行时间。
45. 购物车怎么做？Hash。
46. 签到怎么做？Bitmap。
47. UV 怎么做？HyperLogLog。
48. 布隆过滤器缺点？有误判，不能删除普通元素。
49. 主从复制作用？读扩展和副本。
50. 哨兵作用？监控和故障转移。
51. Cluster 优点？水平扩展。
52. 脑裂风险是什么？网络分区后旧主继续写导致数据冲突。
53. 慢查询怎么查？`SLOWLOG GET`。
54. Redis CPU 高看什么？慢命令、大 key、热 key、Lua、持久化。
55. Redis 内存满怎么办？清理、淘汰、拆 key、扩容。
56. 缓存预热是什么？提前加载热点。
57. 缓存降级是什么？Redis 异常时返回旧值或默认值。
58. 本地缓存优点？抗热点、低延迟。
59. 本地缓存风险？多实例一致性差。
60. key 怎么命名？业务前缀、版本、租户、主键。
61. TTL 怎么设计？按实时性，加随机值。
62. 逻辑过期是什么？value 内带过期时间，异步刷新。
63. Redis 做队列缺点？可靠性和运维能力弱于 MQ。
64. Stream pending 是什么？已投递未 ACK 的消息。
65. Redis 和 Memcached 区别？Redis 结构和持久化能力更强。
66. Redis 安全怎么做？内网、ACL、密码、禁危险命令。
67. 为什么禁 swap？swap 会导致延迟暴涨。
68. fork 有什么影响？数据大时可能造成延迟抖动。
69. 如何防重复领取？Redis SETNX + DB 唯一索引。
70. 库存预扣怎么做？Lua 判断库存再扣减。
71. 缓存命中率低怎么办？分析访问模式、TTL、预热、缓存粒度。
72. 缓存污染是什么？大量冷数据挤出热点。
73. 如何避免污染？短 TTL、LFU、不缓存低价值查询。
74. Redis 适合强一致吗？不适合单独承担强一致事务。
75. Redis 能存大文件吗？不建议。
76. 如何查 key 分布？scan 抽样、bigkeys、监控。
77. 如何处理缓存删除失败？重试队列或 binlog 订阅补偿。
78. 为什么要设置连接超时？避免 Redis 故障拖垮应用 worker。
79. Redis 分布式锁可靠性边界？网络分区、超时、时钟和业务执行时间都可能影响。
80. Redis 面试如何收尾？强调数据结构、缓存一致性、高可用、排障和业务幂等闭环。
